# Primer

## Technical Architecture Specification for Codex-Driven Implementation

**Document version:** 1.0
**Status:** Draft for implementation
**Primary use:** Repository bootstrap and Codex-guided build execution
**Target platforms:** iOS, Android, Web
**Architecture style:** Modular monorepo with app clients, shared packages, and backend services

---

# 1. Purpose

This document defines the technical architecture for building **Primer**, an AI-powered adaptive tutoring application for children.

This version is intentionally written to be used as a **Codex implementation source document**. It is opinionated, concrete, and biased toward getting a buildable repository started quickly.

This document should be copied into the repository as the basis for:

* `Prompt.md`
* `Plan.md`
* `Implement.md`
* `Documentation.md`
* `AGENTS.md`

It also defines the preferred monorepo structure, technology stack, domain boundaries, data model, service interfaces, quality gates, and milestone plan.

---

# 2. Build Objective

Build an MVP of Primer with the following capabilities:

1. Parent account creation and household management.
2. Child profile creation and baseline assessment.
3. Child mobile/web learning experience with:

   * daily sessions
   * live tutoring interaction
   * story mode
   * homework help via text and image upload
4. Parent dashboard with progress and safety review.
5. AI tutoring orchestration layer with safety gating.
6. Persistent learner model and curriculum graph.
7. Audit logging and reporting foundation.

The app is a **tutoring system**, not a general chatbot.

---

# 3. Architectural Principles

## 3.1 Controlled AI, not freeform AI

LLM output must always be constrained by:

* learner state
* curriculum target
* safety policy
* interaction mode policy

## 3.2 Mobile-first, web-supported

The learner experience is mobile-first. Parent dashboard is responsive web first and may also be embedded in mobile.

## 3.3 Monorepo from day one

Use a monorepo so Codex can work across app, API, shared types, validation, and infrastructure with one project root.

## 3.4 Type-safe boundaries

Use shared TypeScript types, runtime validation, and generated contracts wherever practical.

## 3.5 Eventful and auditable

Major actions must emit structured events for:

* learner state changes
* lesson completion
* safety interventions
* settings changes

## 3.6 Prefer simple infrastructure first

Start with a modular monolith backend and clear internal service boundaries. Do not split into microservices at MVP unless scaling forces it.

# 4. Recommended Technology Stack

## 4.1 Monorepo and build tooling

* **Monorepo:** Turborepo
* **Package manager:** pnpm
* **Language:** TypeScript
* **Runtime:** Node.js LTS

## 4.2 Frontend

### Learner mobile app

* **Framework:** React Native with Expo
* **Navigation:** Expo Router
* **State:** Zustand or Redux Toolkit
* **Forms:** React Hook Form + Zod
* **UI:** Tamagui or NativeWind + shared design tokens
* **Media:** Expo AV / camera / image picker

### Parent web app

* **Framework:** Next.js (App Router)
* **Styling:** Tailwind CSS
* **State/data:** TanStack Query
* **Charts:** Recharts

## 4.3 Backend

* **API layer:** Next.js route handlers or NestJS
* **Recommendation for MVP:** Next.js + server modules for faster repo cohesion
* **Async jobs:** Trigger.dev, BullMQ, or a simple queue abstraction backed by Redis
* **Realtime:** WebSocket or SSE for live tutoring sessions

## 4.4 Data and storage

* **Primary database:** PostgreSQL
* **ORM:** Prisma
* **Cache/session/pubsub:** Redis
* **Blob storage:** S3-compatible object storage
* **Search/log indexing:** postpone until needed; use DB-first querying for MVP

## 4.5 Auth and identity

* **Authentication:** Auth.js / NextAuth or Clerk
* **Authorisation:** custom RBAC at service layer
* **Parent gate:** application-level re-auth or challenge workflow

## 4.6 AI and content pipeline

* **LLM orchestration:** backend service calling OpenAI APIs
* **Primary model for orchestration and code-generation support:** GPT-5 family
* **Realtime voice tutoring:** OpenAI Realtime API integration through backend broker
* **Moderation and policy checks:** dedicated safety service using rule layer plus model classification
* **Structured generation:** schema-validated JSON outputs only

## 4.7 Observability

* **Logs:** structured JSON logs
* **Metrics:** OpenTelemetry + hosted dashboard
* **Errors:** Sentry
* **Product analytics:** PostHog or equivalent

## 4.8 Testing

* **Unit:** Vitest
* **Component:** React Testing Library
* **E2E web:** Playwright
* **E2E mobile:** Detox or Maestro
* **Contract tests:** Vitest + Zod/OpenAPI validation

# 9. Primary API Surface

Prefer a typed API. Two good options:

* **Option A:** REST with OpenAPI + generated client
* **Option B:** tRPC for internal type sharing

For MVP, choose one and do not mix casually.

## Recommended choice

Use **REST + OpenAPI** for clearer future interoperability.

## 9.1 API namespaces

* `/auth/*`
* `/households/*`
* `/children/*`
* `/learner-states/*`
* `/curriculum/*`
* `/sessions/*`
* `/stories/*`
* `/homework/*`
* `/safety/*`
* `/reports/*`

## 9.2 Example endpoints

### POST /households

Create household.

### POST /children

Create child profile.

### GET /children/:id/home

Return learner home payload:

* current streak
* recommended lesson
* pending reviews
* rewards snapshot

### POST /children/:id/baseline-assessment/start

Start baseline.

### POST /sessions

Start tutoring session.
Payload:

* childId
* mode
* subject
* optional curriculumNodeId

### POST /sessions/:id/turns

Submit child turn.
Payload:

* inputType: text | voice | image_reference | structured_response
* content

### POST /sessions/:id/complete

Complete session and persist summary.

### POST /homework/parse

Upload or reference image and return extracted problem structure.

### GET /reports/children/:id/progress

Return parent dashboard progress data.

### GET /safety/events

List safety events by household.

### POST /safety/events/:id/review

Mark reviewed.

# 10. AI Orchestration Design

## 10.1 AI gateway rule

No frontend app talks directly to the model provider.
All AI traffic goes through backend orchestration.

## 10.2 AI subcomponents

### Tutor Orchestrator

Inputs:

* learner state
* curriculum target
* session mode
* recent transcript window
* safety context

Outputs:

* child-facing structured response
* state updates
* analytics events

### Content Generator

Produces:

* lesson variants
* story segments
* quizzes
* examples

Must emit schema-valid JSON only.

### Safety Classifier

Classifies:

* restricted topic risk
* emotional dependency language risk
* age-band mismatch
* unsafe instruction risk

### Summariser

Produces:

* session summary
* parent-visible summary
* learner-state delta suggestion

## 10.3 Required structured outputs

Every AI response must conform to an explicit schema.

Example tutor response schema:

```ts
export const TutorResponseSchema = z.object({
  message: z.string(),
  messageStyle: z.enum(["coach", "story", "question", "feedback"]),
  suggestedUi: z.enum(["chat", "mcq", "free_text", "tap_choice", "story_panel"]),
  hints: z.array(z.string()).default([]),
  expectedResponseType: z.enum(["text", "choice", "number", "reading", "none"]),
  masterySignal: z.enum(["unknown", "improving", "mastered", "struggling"]).default("unknown"),
  shouldEscalateSafety: z.boolean().default(false),
  lessonStatePatch: z.record(z.any()).default({})
});
```

## 10.4 Session loop

1. Load child + learner state.
2. Resolve lesson target.
3. Build orchestration input object.
4. Generate candidate response.
5. Run safety validation.
6. If invalid, regenerate or fall back.
7. Persist response and turn metadata.
8. After child reply, update learner model.
9. Repeat until completion or interruption.

# 11. Safety Architecture

## 11.1 Safety layers

Use multiple layers:

### Layer 1: Input classification

Classify incoming child text/image intent.

### Layer 2: Policy rules

Hard rules for forbidden content and routing.

### Layer 3: Model-based output review

Review candidate tutor response before delivery.

### Layer 4: Escalation workflow

Create safety event and notify adult workflow if needed.

## 11.2 Safety event severities

* `info`
* `warning`
* `high`
* `critical`

## 11.3 Prohibited outputs

Do not allow output that:

* encourages secrecy
* claims exclusive emotional bond
* gives self-harm instructions
* gives sexual content
* gives dangerous instructions
* gives diagnosis-like statements

# 20. Definition of Done

The MVP is done when:

1. A parent can sign up and create a household.
2. A child can be created and onboarded.
3. Baseline assessment sets learner state.
4. The child app can run a complete tutoring session.
5. Parent dashboard displays progress and recent sessions.
6. Story mode works.
7. Homework help via uploaded image works in guided mode.
8. Safety events are generated and reviewable.
9. Core commands pass in CI.
10. The repository can be understood and extended by Codex with the included docs.

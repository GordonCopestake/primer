# PROJECT SPECIFICATION: PRIMER

## 1. Document Control

**Project:** Primer
**Version:** 3.0
**Status:** Revised Draft for Implementation
**Primary Platform:** Web app (desktop, tablet, modern mobile browser)
**Architecture Style:** Cloud-AI, local-data, bounded-runtime adaptive learning app
**Primary UX Model:** Full-screen working canvas with hidden chrome
**Primary AI Provider Strategy:** OpenRouter from day one via server-side relay

---

## 2. Product Summary

**Primer** is an adaptive learning web app for any human learner.

Its long-term purpose is to allow a learner to begin at their actual level, including zero literacy, and progress through structured learning toward reading, writing, mathematics, science, physics, and other advanced subjects.

Primer is **not** a child-only product and **not** an adult-only product. It is designed for universal, capability-adaptive learning.

Primer is also **not** a local-inference-first product. Learner data can remain local and the app can avoid logins, but the learning intelligence, scene planning, and rich content generation should assume **cloud AI models are available from the start**.

That means the product must be described honestly as:

> A **cloud-AI adaptive learning app** with **local learner storage** and **no required login**.

The core design goals are:

1. **Full-screen, immersive learning canvas**
2. **Minimal visible UI chrome in normal use**
3. **Bounded scene runtime controlled by the app**
4. **Cloud AI accessed through a secure relay**
5. **Local learner storage with no required account**
6. **Mixed-age-safe default content**

---

## 3. Product Principles

### 3.1 Capability-first, not age-first

Primer adapts primarily to:

* literacy level
* domain mastery
* reasoning ability
* communication style
* pacing
* sensory or motor constraints where configured

Age may matter for content safety or tone selection, but it is not the primary organising principle.

### 3.2 Full-screen working canvas

In normal use, the learner should see the task, scene, or explanation itself, not a traditional app UI.

The application should feel like a responsive learning space, not a dashboard.

### 3.3 Hidden chrome by default

Most UI chrome should remain hidden during normal use and be accessible through a single entry point such as a hamburger menu in the top-left corner.

The working canvas should dominate the screen.

### 3.4 Orb-first interaction metaphor

The AI presence should be represented by a central or near-central **glowing orb**.

The orb is the primary conversational affordance:

* when the AI is speaking, the orb visually indicates speech state
* when idle, the orb remains present but calm
* pressing the orb lets the user talk to the AI
* the orb is a primary interaction element, not decorative fluff

### 3.5 Cloud AI from day one

The product should assume cloud AI models are available from the start. Local device inference may be added later as an enhancement, but it is not a foundational assumption.

### 3.6 Deterministic application control

The model may propose scenes, narration, examples, and state suggestions, but the application owns:

* persistent storage
* scene validation
* permissions
* safety rules
* navigation shell
* content policy
* irreversible actions
* fallback behaviour

### 3.7 Honest privacy posture

The app may store learner data locally and require no login, but it must not claim:

* “fully local”
* “100% private”
* “anonymous”
* “no data leaves the device”

if cloud AI is being used.

### 3.8 Mixed-age-safe content floor

Default content must be safe for mixed-age access.

The product may later introduce age-adaptive content filters, but the default baseline must avoid offensive, sexual, graphic, or clearly adult-coded content.

## 4. Product Vision vs Delivery Scope

## 4.1 Long-Term Product Vision

Primer should eventually support learning across a wide ladder, including:

* zero literacy
* reading
* writing
* numeracy
* mathematics
* science
* physics
* later structured domains

The learner should be able to enter at any level.

## 4.2 v1 Delivery Scope

Primer v1 focuses on proving the architecture and learner experience in a narrower but real band.

### v1 includes

* full-screen immersive canvas UI
* hidden chrome behind menu
* glowing orb interaction model
* local learner storage with no login
* cloud AI access through OpenRouter relay
* deterministic bounded scene runtime
* embedded baseline assessment
* early literacy stages 0–3
* early writing mechanics
* early numeracy foundations
* tap-choice, repeat-sound, trace-symbol, observe scenes
* optional simple read/respond scenes if stable
* persistent learner progress stored locally
* local export/import backup
* admin/settings accessible via hidden chrome

## 4.3 Explicit Non-Goals for v1

The following are not part of v1:

* full advanced STEM tutoring
* full mathematics-to-physics curriculum
* multiplayer or social learning
* teacher dashboards
* cloud sync accounts
* arbitrary model-generated UI
* broad offline tutoring without cloud AI
* unrestricted agent autonomy
* plugin marketplace
* user-authenticated profiles

## 4.4 Later Scope

Later phases may add:

* deeper reading comprehension
* structured writing
* arithmetic fluency
* algebra and symbolic mathematics
* diagram-based science content
* physics problem-solving
* local inference acceleration
* multi-device sync
* optional accounts

## 5. Core UX Model

## 5.1 Experience Summary

Primer should feel like:

* a living canvas
* a guided space
* an AI-mediated educational environment

Primer should **not** feel like:

* a form-based app
* a dashboard with visible controls everywhere
* a learning management system
* a chatbot transcript with a little decoration

## 5.2 Visual Hierarchy

Normal view should contain:

1. **The working canvas**
2. **The orb**
3. **Minimal contextual affordances**
4. **Hidden chrome entry point**

The working canvas always has priority over menus, settings, or scaffolding UI.

## 5.3 Hidden Chrome Rules

Most controls should be hidden unless summoned.

Visible-by-default controls should be minimal:

* hamburger menu, top-left
* orb
* possibly a subtle replay/interrupt affordance near orb
* emergency back/fallback affordance if needed

The hamburger menu opens a chrome layer or side sheet exposing:

* settings
* progress
* export/import
* help
* model/network status
* admin controls
* reset options

## 5.4 Orb Behaviour

The orb is the emotional and interaction anchor of the app.

### Orb states

* `idle`
* `listening`
* `thinking`
* `speaking`
* `error`
* `muted`

### Orb rules

* the orb must be visible during normal learning flow
* pressing or holding the orb initiates voice interaction where available
* tapping the orb during speech may interrupt playback
* the orb must visually animate to reflect current state
* the orb must not obstruct core learning content
* the orb must feel responsive and alive, but not kitschy or childish by default

### Orb meaning

* speaking state = AI currently narrating or explaining
* listening state = app actively capturing learner speech
* thinking state = app is awaiting model output
* idle state = ready
* error state = relay or audio failure
* muted state = sound disabled or unavailable

## 7. Architecture Decision

## 7.1 Honest Architecture Position

Primer is **not** a local-first inference app.

Primer is:

* **cloud-AI for content and guidance**
* **local storage for learner data**
* **no-login by default**
* **deterministic client runtime**

That distinction matters.

## 7.2 Trust Boundaries

The trust boundaries are:

### The client app owns

* learner data storage
* scene rendering
* scene validation
* UI shell
* menu system
* orb state machine
* local progress persistence
* safety enforcement
* fallback behaviour

### The server relay owns

* OpenRouter authentication
* model routing
* request redaction
* provider abstraction
* schema enforcement
* moderation
* logging policy
* rate limiting

### The model does not own

* UI layout
* storage
* direct persistence
* permissions
* navigation shell
* admin controls
* irreversible actions

## 9. Primary Technical Stack

## 9.1 Frontend

* **Framework:** Next.js (App Router) with TypeScript
* **Styling:** Tailwind CSS
* **Animation:** Framer Motion
* **State layer:** Zustand or equivalent
* **Local persistence:** localforage over IndexedDB
* **Canvas layer:** React canvas/SVG/HTML composition as needed
* **Audio capture:** browser media APIs
* **Speech playback:** browser-native TTS initially, richer TTS later if needed

## 9.2 Backend Relay

* **Framework:** Next.js route handlers or separate Node service
* **Model gateway:** OpenRouter from day one
* **Validation:** Zod
* **Moderation:** server-side safety checks before or after model response depending on route
* **Auth:** OpenRouter API key stored server-side only

## 9.3 Storage

* **Learner state:** local browser storage
* **Cached assets:** local browser storage or IndexedDB blobs
* **No required login**
* **No required user account**
* **No cloud learner profile in v1**

## 10. Operating Modes

## 10.1 Normal Mode

The app uses cloud AI models and local learner storage.

This is the default mode.

## 10.2 Reduced Mode

If cloud AI is unavailable:

* the app may still open
* local learner data remains readable
* the app may present limited fallback scenes
* the app must clearly indicate reduced capability

The product should not pretend full tutoring still works without cloud AI.

## 10.3 Developer Mode

Developer mode may expose:

* schema debug
* scene JSON preview
* relay latency
* model IDs
* moderation results
* local state inspection

Developer mode must not be visible in normal learner use.

## 12. Learner Data Model

The learner model should be capability-based, not age-banded.

```ts id="3n1296"
export interface PrimerState {

  learnerProfile: {
    learnerId: string;
    locale: string;

  };

  pedagogicalState: {
    literacyStage: 0 | 1 | 2 | 3 | 4 | 5 | 6;
    domainStage: {
      reading: number;
      writing: number;
      numeracy: number;
      mathematics: number;
      science: number;
      physics: number;

    mastery: {
      phonemes: Record<string, number>;
      graphemes: Record<string, number>;
      vocabularyThemes: Record<string, number>;
      numeracyConcepts: Record<string, number>;
      mathConcepts: Record<string, number>;
      scienceConcepts: Record<string, number>;
    };
  };

  runtimeSession: {
    activeSceneId: string | null;
    turnCount: number;
    runningSummary: string | null;
    lastTurns: { role: "user" | "assistant"; content: string }[];
    pendingAssetJobs: string[];
  };

  settings: {
    soundEnabled: boolean;
    captionsEnabled: boolean;
    imageGenerationEnabled: boolean;
    adminPinEnabled: boolean;
    microphoneEnabled: boolean;
  };

  uiState: {
    chromeOpen: boolean;
    orbState: "idle" | "listening" | "thinking" | "speaking" | "error" | "muted";
    currentViewMode: "immersive" | "menu" | "admin";
  };

  cache: {
    imageAssets: Record<string, string>;
    audioAssets: Record<string, string>;
  };

```

## 12.1 State Rules

* `schemaVersion` is mandatory
* learner state persists locally
* `lastTurns` is bounded
* `runningSummary` is a compressed context summary for cloud calls
* no login is required to create or load local learner state
* local state must survive reloads and normal browser restarts where storage is available

## 14. Curriculum and Mastery Engine

The curriculum engine is deterministic and local.

## 14.1 Responsibilities

The engine owns:

* embedded baseline assessment
* objective selection
* progression rules
* regression/review rules
* mastery updates
* scene eligibility
* subject unlocking
* transition from concrete to abstract content
* guardrails on what the Director is allowed to propose

## 14.2 Non-Responsibilities

The engine does not:

* render UI
* directly call OpenRouter
* generate final narration text
* generate final visual assets

## 14.3 Output Contract

type CurriculumDecision = {

## 15. Embedded Baseline Assessment

Primer should use **embedded baseline assessment** rather than blunt self-rating.

## 15.1 Purpose

The goal is to infer the learner’s starting point through interaction.

## 15.2 Examples

Depending on current domain:

* matching sound to picture
* tracing a symbol
* identifying a written token
* choosing the larger quantity
* responding to a very short written or spoken prompt
* demonstrating simple prerequisite reasoning

## 15.3 Rules

* baseline assessment should feel like guided interaction
* the Director may suggest assessment scenes
* the curriculum engine decides when assessment is complete

## 16.1 Default Screen Layout

Normal layout should prioritise:

* working canvas occupying nearly all available space
* orb visible over the scene
* subtle hamburger icon in the top-left

The hamburger menu opens a panel or overlay exposing:

* progress

* sound/captions
* export/import
* reset options
* help/about
* admin controls
* network/model status

The menu should not be permanently visible in immersive mode.

## 16.3 Canvas Rules

The working canvas should support:

* visual scenes
* text where appropriate to learner level
* tracing surfaces
* symbol interaction
* image-backed choice scenes
* explanatory content blocks
* animated transitions

## 17. Orb Interaction Specification

## 17.1 Orb Purpose

The orb is:

* the visible embodiment of the AI presence
* the user’s main speech interaction button
* the primary state indicator for AI activity

## 17.2 Orb Interaction Rules

* tap orb to start listening
* tap orb during speech to interrupt
* tap orb while idle to invite conversation
* orb must show immediate visual feedback
* long press may optionally expose advanced voice mode later
* keyboard and accessibility alternatives must exist

## 17.3 Orb Animation Rules

The orb should animate differently by state:

* `idle`: soft pulse
* `listening`: brighter reactive pulse
* `thinking`: slow rotating or breathing animation
* `speaking`: waveform-like energy pulse
* `error`: restrained warning state, not alarming
* `muted`: subdued dim state

## 17.4 Orb Placement

Preferred placement:

* lower centre
* lower right
* or another non-obstructive location

It must not obscure core content or frequent touch targets.

The interpreter must:

* reject invalid payloads
* reject unsupported scene or interaction types
* reject HTML or arbitrary code
* reject scenes that exceed narration budget
* reject unsafe content
* fall back to a safe local scene if validation fails

## 19.3 v1 Scene Budgets

* one primary interaction per scene
* one dominant visual focus
* maximum four tap-choice options
* narration length must be bounded

## 20. Director and Chat Roles

The product should separate **Director** and **Conversational Chat** roles conceptually, even if they may sometimes use the same underlying model.

## 20.1 Director Role

The Director proposes the next scene.

It may:

* select a scene shape
* write narration
* choose examples
* shape tone
* suggest visuals
* suggest state deltas

It may not:

* directly mutate stored state
* bypass validation
* bypass content policy
* create arbitrary UI

## 20.2 Chat Role

The conversational model supports direct learner conversation when the user activates the orb.

It may:

* answer learner questions
* explain concepts
* encourage reflection
* clarify tasks
* elaborate on current content

It must remain bounded by:

* current domain/context
* safety rules
* mixed-age-safe tone
* no inappropriate storyline generation

## 22. OpenRouter Integration Rules

## 22.1 Required from Start

OpenRouter integration is required from the start of implementation.

Do not design the repo around a hypothetical future cloud layer. Build it in immediately.

## 22.2 Model Selection Rules

* models must be configured centrally
* exact model IDs must be configurable via env
* Director, Chat, Vision, and Image routes must be separately configurable
* model fallback logic may be added later but is not required in the first pass

## 22.3 Security Rules

* OpenRouter API key must never be exposed client-side
* all calls must go through server routes
* request logging must be minimal
* prompt content should be reduced to minimum useful context

---

## 23. Audio Interaction Model

## 23.1 Voice Flow

1. User presses orb
2. Orb enters listening state
3. Browser captures audio
4. Audio is transcribed or sent for processing depending on implementation choice
5. Server returns conversational response and/or suggested next scene
6. Orb enters speaking state
7. Reply is spoken and/or displayed
8. Scene updates if needed

## 23.2 Minimum Viable Audio Support

v1 does not need perfect voice stack quality, but it must support a stable basic loop:

* press orb
* speak
* AI replies
* learner sees or hears result

If browser/device constraints make voice unreliable, the app must degrade gracefully to text input or tap interaction.

---

## 24. Persistence and No-Login Model

## 24.1 Storage Policy

Learner data is stored locally by default.

This includes:

* learner profile
* progress
* current objective
* recent context summary
* settings
* cached assets

## 24.2 No Required Login

v1 should require no login and no account creation.

The app should be able to start immediately for a new learner.

## 24.3 Export / Import

The app must support:

* export of learner state
* import of learner state
* reset of local learner state

This is required because there is no default cloud account system.

---

## 25. Error Handling and Fallback

## 25.1 Failure Classes

* no network
* OpenRouter unavailable
* server relay timeout
* malformed model output
* image generation timeout
* local storage unavailable
* microphone unavailable
* speech capture failure
* unsafe content detected
* schema validation failure

## 25.2 Fallback Behaviour

For all failures:

* preserve local learner data where possible
* avoid blank screens
* keep orb and menu responsive
* present a safe fallback scene or message
* be honest when cloud AI is unavailable

## 25.3 Safe Fallback Example

Fallback scene may contain:

* calm ambient background
* short message such as: “I can’t reach the learning service right now. You can still review what you’ve done.”
* menu access
* reviewable local content if available

---

## 26. Accessibility

Primer must support:

* mouse/touch interaction
* keyboard navigation
* captions/subtitles for spoken output
* large touch targets for orb and key controls
* high-contrast compatible rendering
* non-voice alternatives when microphone use is not possible
* non-speaking learner paths via tap and symbol interaction

## 29. Acceptance Criteria for v1

Primer v1 is complete when:

* the learner sees a full-screen immersive canvas in normal use
* most chrome is hidden behind a hamburger menu
* the orb is visible and acts as the primary AI interaction control
* pressing the orb allows the learner to talk to the AI
* the app uses OpenRouter via server-side relay from day one
* learner state persists locally with no login required
* the Director can propose validated scenes
* the Chat route can respond to learner questions
* invalid model output cannot break the UI
* the app supports early literacy and early numeracy scenes
* image generation is optional and non-blocking
* the app remains usable during network/model failures
* default content remains mixed-age-safe

---

## 30. Final Implementation Notes for Codex

Codex should treat the following as non-negotiable:

1. **Full-screen immersive canvas is the default UI.**
2. **Most chrome stays hidden behind a hamburger menu.**
3. **The glowing orb is a primary interaction control, not decoration.**
4. **Cloud AI via OpenRouter is required from the start.**
5. **Learner data is local, with no required login.**
6. **The app owns the runtime and validation; the model does not own the UI.**
7. **Default content must be mixed-age-safe.**
8. **v1 is still narrower than the full long-term vision.**

## 30.1 Anti-Goals for Codex

Codex must not:

* build a chrome-heavy dashboard UI
* expose the OpenRouter key in the client
* rely on arbitrary model-generated HTML
* make login mandatory
* pretend the app works fully offline
* make the orb secondary or purely decorative
* hardcode childish tone across the app
* hardcode adult-only framing either

---

## 31. Governing Product Definition

Primer is a cloud-AI adaptive learning app with local learner storage, no required login, a full-screen working canvas, hidden chrome, and an orb-centred conversational interface.

That definition should govern implementation.

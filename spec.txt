# PROJECT SPECIFICATION: PRIMER

## 1. Document Control

**Project:** Primer
**Version:** 2.2
**Status:** Revised Draft for Implementation
**Primary Platform:** Installable web app (desktop, tablet, modern mobile browser)
**Architecture Style:** Static local-first PWA with optional cloud relay
**Product Goal:** Universal adaptive learning from zero literacy to advanced subjects over time

---

## 2. Product Summary

**Primer** is a local-first, installable adaptive learning web app for any human learner.

Its long-term purpose is to allow a learner to begin at their actual level, including zero literacy, and progress through structured learning toward reading, writing, numeracy, mathematics, science, physics, and other advanced subjects.

Primer is **not** designed specifically for children or specifically for adults. It is designed for **capability-adaptive learning** across a wide learner range.

The core runtime, learner state, progression logic, and fallback learning loop live on-device. Optional cloud services may enrich narration, dialogue, imagery, explanation depth, and ambiguous interpretation, but cloud inference is never required for the core learning loop.

Primer is intentionally **not** a model-controlled application. It is a deterministic application with bounded AI augmentation.

### Core Product Positioning

Primer should be described as:

> A **local-first adaptive learning app** with **optional cloud augmentation**.

Primer should not be described as:

* a child-only product
* an adult-only product
* a fully model-controlled tutor
* “100% local” unless all inference is local
* “anonymous” unless that is actually true end-to-end

## 3. Product Principles

### 3.1 Learner-universal, not age-targeted

Primer is for any human learner. The system must adapt to literacy level, subject mastery, communication mode, pace, and reasoning ability rather than assume a particular age group.

### 3.2 Capability-first adaptation

The app adapts primarily to:

* literacy level
* subject mastery
* communication modality
* confidence and pacing
* sensory or motor constraints where configured or detected
* learner preference for explanation style where configured

### 3.3 Deterministic progression

The local runtime, not the model, owns:

* objective selection
* prerequisite enforcement
* mastery scoring
* assessment completion
* state persistence
* permissions
* safety rules
* irreversible actions
* lesson unlocking
* fallback behaviour

### 3.4 Bounded AI control

AI may shape:

* narration
* examples
* analogies
* presentation tone
* optional narrative continuity
* optional visual suggestions

AI may not own:

* progression state
* mastery state
* security
* local storage
* navigation shell
* irreversible actions
* policy decisions

### 3.5 Neutral-safe content floor

Default content must be safe for mixed-age exposure.

Baseline content must avoid:

* sexual themes
* graphic violence
* offensive stereotypes
* humiliating or degrading framing
* partisan political persuasion
* age-coded exclusionary assumptions
* storylines likely to be inappropriate for children

This is **not** because the product is child-specific. It is because mixed-age-safe content is the safest universal default.

### 3.6 Progressive abstraction

Primer must support progression from:

* pre-literacy and concrete recognition
* phoneme and symbol mapping
* reading and writing
* arithmetic and structured reasoning
* symbolic manipulation
* abstract mathematical and scientific concepts

### 3.7 Graceful degradation

If local inference, cloud services, storage, or asset generation fail, the app must fall back to a known-safe local path and remain usable.

## 4. Product Vision vs Delivery Scope

## 4.1 Long-Term Product Vision

Primer is intended to support a full learning ladder from zero literacy to advanced domains, including:

* reading
* writing
* numeracy
* mathematics
* science
* physics
* additional structured subjects later

The architecture must support learners entering at any point on that ladder.

## 4.2 v1 Delivery Scope

Primer v1 must **not** attempt to fully solve the entire ladder.

Primer v1 focuses on proving the adaptive runtime and deterministic learning architecture in a narrower but meaningful band.

### v1 Scope

* installable PWA shell
* local persistence of learner state
* deterministic curriculum and mastery engine
* embedded baseline assessment
* literacy stages 0–3
* early writing mechanics
* early numeracy foundations
* bounded scene interpreter
* local TTS where supported
* local STT where supported
* tap-first and replay-based fallback interactions
* optional cloud Director for scene wording and presentation
* optional non-blocking cloud image generation
* local admin/settings controls
* export/import of learner data

## 4.3 Explicit Non-Goals for v1

The following are not part of v1:

* full advanced STEM tutoring
* full algebra-to-physics progression
* unrestricted open-ended agent autonomy
* teacher dashboards
* multiplayer/social learning
* multi-user cloud sync
* cloud-dependent core interaction
* arbitrary model-generated HTML
* uncontrolled free-form browsing
* broad exam-prep curriculum
* behavioural growth/engagement mechanics

## 4.4 Planned Later Scope

Later phases may add:

* reading comprehension
* structured writing
* arithmetic fluency
* pre-algebra
* formal mathematics scenes
* diagram-based reasoning
* science concept scaffolding
* physics problem-solving
* broader subject packs

## 5. Learner Experience Model

## 5.1 Interaction Model

Primer presents a **scene**, not a conventional app page.

Each scene has:

* one pedagogical objective
* one primary interaction mode
* concise narration or explanation
* one primary visual or symbolic focus
* a predictable completion or fallback path

## 5.2 Tone Requirements

The system tone must be:

* respectful
* clear
* encouraging without flattery spam
* non-patronising
* non-age-coded
* emotionally safe
* neutral enough for mixed-age access

The app must not assume that simple learning must feel childish, and must not assume that advanced learning must feel adult-coded.

## 5.3 Content Framing Requirements

Acceptable default content framing includes:

* nature
* shapes
* patterns
* journeys
* building
* puzzles
* motion
* sound
* stars
* maps
* tools
* machines
* exploration
* neutral storybook scenarios
* abstract symbolic scenes

Unacceptable default content framing includes:

* romance
* sexual themes
* alcohol/drug culture
* graphic conflict
* humiliation comedy
* partisan politics
* frightening horror
* age-shaming
* intelligence-shaming
* “adult reward” framing

## 5.4 Fixed Shell vs Flexible Scene

The application always owns a minimal fixed shell. This shell is never model-generated.

The fixed shell includes:

* microphone/listen affordance where available
* stop/interruption control
* replay control
* loading/thinking indicator
* settings/admin entry point
* error recovery state
* safe fallback scene entry

The model controls only bounded scene content within the approved DSL.

## 6. Deployment Model

## 6.1 Primary Deployment Decision

Primer v1 is architected as a **static local-first PWA**.

The app must be deployable as:

* a static web app
* an installable PWA
* a local-first app with no required backend for the core learning loop

## 6.2 Optional Cloud Relay

Cloud-enabled features must not be structurally fused into the client runtime.

Optional remote features must be routed through a separate **relay service**.

The relay may be:

* self-hosted
* vendor-hosted
* disabled entirely

The relay is responsible for:

* provider authentication
* prompt minimisation
* request redaction
* schema enforcement
* moderation
* model pinning
* rate limiting
* error normalisation
* disabling request-body logging by default

## 6.3 Operating Modes

Primer supports:

* **local-only**

  * no remote calls
  * core learning loop only

* **local-first assisted**

  * core loop local
  * optional Director and/or image generation enabled

* **development**

  * additional diagnostics
  * fixture content
  * mock relay permitted

## 8. Capability Tiers

The app must classify the current device into a capability tier at startup.

## 8.1 Tier: `minimal`

Available features:

* local persistence where supported
* cached or embedded assets
* observe scenes
* tap-choice scenes
* trace-symbol scenes
* browser-native TTS or prerecorded fallback audio
* no local STT requirement

## 8.2 Tier: `standard-local`

Available features:

* all `minimal` features
* local TTS
* local STT on CPU/WASM where supported
* repeat-sound interactions
* basic local audio processing

## 8.3 Tier: `accelerated-local`

Available features:

* all `standard-local` features
* accelerated local inference where available
* lower-latency audio
* richer local model options
* more aggressive prefetch and caching

## 8.4 Tier Selection Inputs

Capability detection should consider:

* secure context
* worker support
* service worker support
* IndexedDB support
* OPFS support
* microphone availability
* local storage quota where estimable
* hardware acceleration where detectable

## 9. Technical Stack

## 9.1 Client App

* **Language:** TypeScript
* **Recommended shell:** Next.js static export or equivalent static-capable framework
* **Styling:** Tailwind CSS
* **Animation:** Framer Motion
* **PWA:** service worker + manifest
* **State layer:** local store with explicit schema versioning
* **Storage:** IndexedDB for structured state and metadata
* **Blob storage:** OPFS where supported
* **Heavy work isolation:** Web Workers

## 9.2 Local Audio Runtime

* **TTS:** browser-native fallback first, optional higher-quality local runtime-backed TTS where supported
* **STT:** lightweight browser-compatible local speech stack where supported
* **No main-thread heavy inference**

## 9.3 Relay Service

* **Separate deployable service**
* **Pinned provider models**
* **Strict request/response validation**
* **Redaction/minimisation before provider calls**
* **Stable error contracts**

## 9.4 Validation

* **Schema validation:** Zod
* **Scene validation:** strict before render
* **State migration:** explicit versioned migrations
* **Asset validation:** checksum/version manifest

## 11. Domain and Learner Model

The learner model must be organised primarily by capability, mastery, and preferences rather than age band.

## 11.1 State Model

```ts
export interface PrimerState {
  schemaVersion: number;

  learnerProfile: {
    learnerId: string;
    locale: string;
    interests: string[];
    avatarSeed: string | null;
    preferredModalities: Array<"audio" | "visual" | "text" | "symbolic" | "interactive">;
  };

  pedagogicalState: {
    literacyStage: 0 | 1 | 2 | 3 | 4 | 5 | 6;
    domainStage: {
      reading: number;
      writing: number;
      numeracy: number;
      mathematics: number;
      science: number;
      physics: number;
    };
    currentObjectiveId: string | null;
    assessmentStatus: "not-started" | "in-progress" | "complete";
    mastery: {
      phonemes: Record<string, { score: number; seen: number; updatedAt: number }>;
      graphemes: Record<string, { score: number; seen: number; updatedAt: number }>;
      vocabularyThemes: Record<string, { score: number; seen: number; updatedAt: number }>;
      numeracyConcepts: Record<string, { score: number; seen: number; updatedAt: number }>;
      mathConcepts: Record<string, { score: number; seen: number; updatedAt: number }>;
      scienceConcepts: Record<string, { score: number; seen: number; updatedAt: number }>;
    };
  };

  runtimeSession: {
    activeSceneId: string | null;
    recentTurns: { role: "user" | "assistant"; content: string }[];
    runningSummary: string | null;
    pendingAssetJobs: string[];
  };

  consentAndSettings: {
    cloudEnabled: boolean;
    cloudImageEnabled: boolean;
    cloudVisionEnabled: boolean;
    adminPinEnabled: boolean;
    captionsEnabled: boolean;
    soundEnabled: boolean;
    storagePersistenceGranted: boolean | "unknown";
  };

  capabilities: {
    tier: "minimal" | "standard-local" | "accelerated-local";
    webgpu: boolean;
    opfs: boolean;
    indexedDb: boolean;
    localTTS: boolean;
    localSTT: boolean;
    microphone: boolean;
  };

  assetIndex: {
    byId: Record<
      string,
      {
        kind: "image" | "audio" | "model";
        storage: "idb" | "opfs";
        mimeType: string;
        bytes: number;
        checksum: string;
        version: string;
        lastAccessedAt: number;
      }
    >;
  };
}
```

## 11.2 State Rules

* `schemaVersion` is mandatory
* `recentTurns` is bounded and not treated as the durable learner record
* `pedagogicalState` is authoritative for progression
* `runningSummary` is context compression only
* cached assets are disposable and must not be treated as the only durable store

## 14. Embedded Baseline Assessment

The product must use **embedded baseline assessment**, not a blunt “rate yourself” flow.

## 14.1 Purpose

The goal is to infer where the learner should start without assuming prior literacy or prior schooling.

## 14.2 Examples

Depending on enabled domains, baseline assessment may include:

* recognising a sound
* matching sound to picture
* tracing a basic symbol
* selecting a larger quantity
* identifying a simple written token
* following a one-step instruction
* responding to a very short prompt
* demonstrating prerequisite reasoning

## 14.3 Rules

* baseline assessment must feel like interaction, not examination
* the Director may propose assessment scenes but may not mark assessment complete
* completion is decided only by the curriculum engine

## 16. Scene DSL

The model must not output HTML, JSX, CSS, markdown UI, or executable code.

It may only output bounded JSON matching the approved scene contract.

## 16.1 Scene Blueprint

```ts
export type SceneBlueprint = {
  version: 1;
  scene: {
    id: string;
    kind: "intro" | "assessment" | "lesson" | "practice" | "review" | "reward" | "fallback";
    objectiveId: string;
    transition: "fade" | "slide" | "pan";
    tone: "calm" | "encouraging" | "celebratory" | "focused" | "curious";
  };

  narration: {
    text: string;
    maxChars: number;
    estDurationMs: number;
    bargeInAllowed: boolean;
  };

  visualIntent?:
    | {
        type: "cached-asset";
        assetId: string;
      }
    | {
        type: "recipe";
        recipeId: string;
        vars: Record<string, string>;
      };

  interaction:
    | { type: "none" }
    | {
        type: "tap-choice";
        options: {
          id: string;
          audioLabel: string;
          assetId?: string;
          symbol?: string;
        }[];
      }
    | {
        type: "repeat-sound";
        phoneme: string;
      }
    | {
        type: "trace-symbol";
        target: string;
        guideAssetId?: string;
      }
    | {
        type: "read-respond";
        promptText: string;
      }
    | {
        type: "compose-text";
        promptText: string;
        minLength?: number;
      }
    | {
        type: "equation-step";
        expression: string;
        expectedForm?: string;
      }
    | {
        type: "diagram-label";
        diagramAssetId: string;
        targets: string[];
      }
    | {
        type: "explain-back";
        promptText: string;
      };

  evidence?: {
    observedSkill?: string;
    confidenceHint?: number;
  };
};
```

## 16.2 Interpreter Rules

The interpreter must:

* reject invalid schema
* reject unsupported component types
* reject scenes exceeding narration budget
* reject text-heavy scenes above learner capability
* reject overlarge option sets
* reject arbitrary raw HTML
* render only known atomic components
* fall back to a safe local scene on validation failure

## 16.3 Scene Budgets

v1 runtime-enforced budgets:

* maximum one primary interaction
* maximum one visual intent
* maximum four tap choices
* narration length bounded by curriculum decision
* no arbitrary long-form text blocks
* no dynamic layout generation from model output

## 17. Director Contract

The Director is a **scene proposer**, not the application controller and not the curriculum engine.

## 17.1 Director Responsibilities

The Director may:

* choose a scene shape from allowed options
* write concise narration
* maintain emotional continuity
* choose a visual recipe
* supply advisory evidence hints
* choose examples or analogies appropriate to the learner state

## 17.2 Director Prohibitions

The Director must not:

* mutate persistent learner state
* set literacy stage
* mark assessment complete
* unlock subjects
* create unsupported UI
* output HTML or code
* claim consciousness
* imply exclusive friendship
* shame the learner
* use sexual, graphic, or mixed-age-inappropriate framing
* request disallowed cloud operations

## 17.3 Director Prompt Skeleton

```text
You are the Primer Director.

Your role is to propose the next bounded learning scene.
You do not control state, curriculum progression, safety rules, storage, or permissions.
You may only emit JSON matching the supplied SceneBlueprint schema.

You must stay within the provided hard constraints.

Rules:
- Keep narration concise and concrete.
- Stay within the allowed interaction types.
- Do not introduce written text above learner capability.
- Do not output HTML, JSX, markdown UI, or code.
- Do not claim sentience or exclusive relationship.
- Do not use sexual, graphic, humiliating, or age-inappropriate framing.
- Use only allowed visual recipes.
- Provide only advisory evidence hints, not state mutation.
```

## 18. Content and Safety Policy

## 18.1 Default Content Standard

Default content must be suitable for mixed-age access.

This means:

* neutral-safe themes
* non-sexualised examples
* non-graphic imagery
* respectful language
* no humiliating corrective tone
* no adult-coded storyline assumptions

## 18.2 Relationship Safety

Primer must not:

* imply exclusive friendship
* emotionally manipulate dependence
* shame the learner
* infantilise adult learners
* sexualise interaction
* pretend to judge moral worth

## 18.3 Subject Safety

As subject complexity increases:

* tone may become more technical
* abstraction may increase
* explanation density may increase

But the app must still avoid:

* humiliation framing
* ideological persuasion disguised as teaching
* inappropriate mature themes
* “you should already know this” language

## 19. Local Audio Runtime

## 19.1 Requirements

* no heavy audio work on the main thread
* interruption and replay supported
* capability-tier-based behaviour
* no hard dependence on STT in `minimal` tier

## 19.2 TTS Strategy

Priority order:

1. higher-quality local TTS where supported
2. browser-native speech where acceptable
3. prerecorded fallback clips for shell phrases

## 19.3 STT Strategy

Priority order:

1. local STT where supported
2. structured tap-first alternatives
3. no mandatory voice input in `minimal` mode

---

## 20. Local Validation Strategy

Simple tasks must not depend on cloud interpretation.

## 20.1 Local-first Validation

Use local validation for:

* stroke presence
* approximate shape
* bounding-box fit
* occupancy
* directional guidance where relevant
* coarse similarity scoring

## 20.2 Cloud Escalation Rules

Escalate to cloud only when:

* local confidence falls into a defined ambiguity band
* the task is genuinely open-ended
* remote interpretation is enabled
* the curriculum engine allows escalation

---

## 21. Asset and Model Management

Local-first audio and rich offline behaviour require explicit asset and model handling.

## 21.1 Requirements

The app must support:

* versioned model manifests
* checksum validation
* resumable downloads where feasible
* lazy installation
* deletion and re-download
* storage estimation before large downloads
* storage pressure handling
* per-tier asset strategies

## 21.2 Asset Types

* built-in shell assets
* lesson assets
* generated images
* local model packs
* prerecorded fallback audio

## 21.3 Persistence Strategy

* metadata: IndexedDB
* larger blobs: OPFS where supported
* asset manifest tracks size, version, checksum, and last access

## 21.4 Eviction Policy

The app may evict:

* oldest generated assets
* non-essential cached audio
* non-active model packs

The app must not silently evict:

* core learner state
* migration-critical metadata
* only-available fallback assets

---

## 22. Storage and Offline Behaviour

The local-first design depends on web storage and offline primitives such as service workers, persistence requests, and OPFS-backed blob storage. ([MDN Web Docs][2])

## 22.1 Offline Requirements

The app must support:

* startup without network
* persisted learner progress across reloads
* recovery into a safe scene after interruption
* offline operation of the core learning loop

## 22.2 Persistence Flow

During onboarding or settings flow, the app should:

* inspect whether storage is already persistent
* request persistent storage where supported
* estimate available storage before large downloads
* warn before optional large model installs

## 22.3 Backup and Restore

v1 must support:

* export of learner data and settings
* import of local backup
* optional encrypted export if implemented

Backup/export is part of resilience, not a side feature.

All remote APIs are optional and must go through the relay.

## 23.1 `/director`

### Request

```ts
type DirectorRequest = {

  runtimeSummary: string | null;
  latestInput: {
    type: "transcript" | "tap-choice" | "trace-result" | "system-start";
    content: string;
  };
  hardConstraints: {
    activeDomain: LearningDomain;
    literacyStage: 0 | 1 | 2 | 3 | 4 | 5 | 6;
    objectiveId: string | null;
    allowedSceneKinds: Array<
      "assessment" | "lesson" | "practice" | "review" | "reward" | "fallback"
    >;
    allowedInteractionTypes: Array<
      | "none"
      | "tap-choice"
      | "repeat-sound"
      | "trace-symbol"
      | "read-respond"
      | "compose-text"
      | "equation-step"
      | "diagram-label"
      | "explain-back"
    >;
    maxNarrationChars: number;
    imageGenerationAllowed: boolean;
    locale: string;
  };

```

type DirectorResponse = {

### Relay Rules

* validate request shape

## 24. Execution Pipelines

## 24.1 Startup Flow

1. Load app shell
2. Register service worker where supported
3. Detect capability tier
4. Load local state
5. Attempt migrations if needed
6. Check persistence status
7. Restore last safe scene or start new session
8. Run embedded baseline assessment if needed
9. Enter local-only mode automatically if relay is disabled or unavailable

## 24.2 Core Scene Flow

1. Learner action is captured locally
2. Curriculum engine computes hard constraints
3. If Director is enabled, a bounded proposal may be requested
4. Proposal is validated strictly
5. If valid, scene is rendered
6. If invalid, local deterministic fallback scene is used
7. Evidence is recorded locally
8. Mastery state is updated locally

## 24.3 Voice Flow

1. User activates listen
2. Local audio runtime captures speech if supported
3. Transcript is normalised locally
4. Minimal request is prepared if relay use is allowed
5. Director may propose a scene
6. Runtime validates the scene
7. Scene is rendered
8. Narration starts
9. Optional image work is queued in the background

## 24.4 Trace Flow

1. User traces symbol
2. Local validator scores stroke features
3. If confidence is high enough, continue locally
4. If ambiguous and allowed, optionally escalate to `/vision`
5. Curriculum engine updates evidence locally
6. Next scene is rendered

## 25. Privacy and Data Policy

## 25.1 Local Data Policy

By default, Primer stores locally:

* learner profile
* pedagogical progress
* runtime summary
* settings/admin preferences
* cached assets
* model pack metadata

## 25.2 Cloud Processing Policy

If cloud mode is enabled:

* only minimum required context should be sent
* raw learner state should be summarised rather than dumped wholesale
* provider access must go through relay
* request bodies should not be logged by default
* cloud use must be visible in settings/admin UI

## 25.3 Prohibited Claims

The product must not claim:

* “100% local” unless true
* “anonymous” unless true end-to-end
* “nothing ever leaves the device” when cloud mode is enabled
* vague privacy guarantees that the deployment cannot support

## 25.4 Analytics

Default mixed-age-safe mode should avoid:

* ad tech
* behavioural profiling
* manipulative engagement systems

If analytics exist at all, they must be strictly limited and not part of the learning loop.

## 27. Error Handling and Fallback

## 27.1 Failure Classes

* capability insufficient
* local model unavailable
* microphone unavailable
* network unavailable
* relay rate limit
* malformed relay response
* asset generation timeout
* storage quota exceeded
* migration failure
* asset checksum mismatch

## 27.2 Fallback Strategy

For all failures:

* preserve learner progress where possible
* avoid blank screens
* return to a known-safe local scene
* use calm neutral phrasing
* keep the app operable

## 27.3 Safe Fallback Scene

A safe fallback scene must be bundled locally and always available.

It includes:

* simple ambient visual
* short neutral narration
* one safe interaction path
* no dependency on network
* no dependency on heavy local inference

Example narration:

> “Let’s continue another way.”

## 28. Accessibility Requirements

Primer must not assume that all learners are comfortable with speech, hearing, tracing, or dense text.

v1 should support:

* tap-first alternatives when voice is unavailable
* caption option for narration
* replay for spoken output
* high-contrast compatible UI
* motor-tolerant tracing thresholds
* non-speaking learner path using choices and tracing
* no dependence on tiny targets

Later phases may add:

* richer keyboard-first flows
* screen-reader-specific optimisation for advanced content
* symbolic accessibility layers for maths/science content

## 33. Codex Implementation Notes

Codex should treat the following as non-negotiable architectural constraints:

1. **The curriculum engine is deterministic and local.**
2. **The app is a static local-first PWA.**
3. **Cloud is optional and relay-mediated.**
4. **The Director is a bounded scene proposer, not a controller.**
5. **Default content must be mixed-age-safe.**
6. **v1 is limited to early literacy and early numeracy despite the broader long-term architecture.**

Codex should prioritise work in this order:

1. local state + migrations
2. capability detection
3. fallback scene
4. curriculum engine
5. scene schemas + interpreter
6. local audio runtime
7. v1 interaction components
8. relay endpoints
9. asset/model management
10. admin/settings
11. hardening

* rely on server-only Next.js features for the core loop
* block scene rendering on cloud image generation
* let cloud responses bypass schema validation
* assume v1 includes advanced STEM scenes
* hardcode child-only branding or adult-only framing

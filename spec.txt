# Primer

## Technical Architecture Specification for Codex-Driven Implementation

**Document version:** 1.1
**Status:** Draft for implementation
**Primary use:** Repository bootstrap and Codex-guided build execution
**Target platforms:** iOS, Android, Web
**Architecture style:** Modular monorepo with universal learner app, parent web app, shared packages, and a separate API service

---

# 1. Purpose

This document defines the technical architecture for building **Primer**, an AI-powered adaptive tutoring application for children.

This version is written to act as an **implementation source document** for Codex and human developers. It is opinionated, concrete, and designed to reduce architectural drift during repository bootstrap and milestone delivery.

This document is intended to be copied into the repository and used as the basis for:

* `Prompt.md`
* `Plan.md`
* `Implement.md`
* `Documentation.md`
* `AGENTS.md`

It defines the preferred monorepo structure, technology stack, domain boundaries, data model, service interfaces, quality gates, privacy and safety controls, and milestone plan.

---

# 2. Product Objective

Build an MVP of Primer with the following capabilities:

1. Parent account creation and household management.
2. Child profile creation and baseline assessment.
3. Child learning experience on iOS, Android, and Web with:

   * daily sessions
   * guided tutoring interaction
   * story mode
   * homework help via text and image upload
4. Parent dashboard with progress, transparency, and safety review.
5. AI tutoring orchestration layer with safety gating.
6. Persistent learner model and curriculum graph.
7. Audit logging, privacy controls, and reporting foundation.

Primer is a **tutoring system**, not a general chatbot, social companion, or open-ended assistant.

# 3. Architectural Principles

## 3.1 Controlled AI, not freeform AI

LLM output must always be constrained by:

* learner state
* curriculum target
* safety policy
* interaction mode policy
* parent-controlled permissions
* privacy policy

## 3.2 Local-first where practical, not ideological

Primer shall prefer local inference where it is appropriate, performant, and safe.

Cloud inference shall be used when:

* the task exceeds local model capability,
* the device cannot run the local model at acceptable latency,
* the feature requires stronger reasoning or multimodal processing,
* policy requires server-side review,
* the browser environment cannot support the required model/runtime,
* the local model is unavailable, unhealthy, or disabled.

Local-first is an optimisation and privacy improvement, not an absolute requirement.

## 3.3 Universal learner app, web-first parent app

The learner experience is delivered through a **single universal app** supporting:

* iOS
* Android
* Web

The parent dashboard is a separate **responsive web application**.

## 3.4 Monorepo from day one

Use a monorepo so Codex can work across app, API, shared types, validation, and infrastructure from one root.

## 3.5 Type-safe boundaries

Use shared TypeScript types, runtime validation, explicit contracts, and generated clients wherever practical.

## 3.6 Eventful and auditable

Major actions must emit structured events for:

* learner state changes
* session lifecycle
* curriculum progression
* safety interventions
* settings changes
* consent changes
* inference and routing decisions

## 3.7 Simple operational shape first

Start with a **modular monolith API service** and clear internal module boundaries. Do not split into microservices during MVP.

## 3.8 Privacy and child protection by design

The system must minimise data collection, retain only necessary data, provide parent visibility and control, and maintain explicit records of consent, policy decisions, and review actions.

# 4. Recommended Technology Stack

## 4.1 Monorepo and build tooling

* **Monorepo:** Turborepo
* **Package manager:** pnpm
* **Language:** TypeScript
* **Runtime:** Node.js LTS

## 4.2 Frontend

### Learner universal app

* **Framework:** React Native with Expo
* **Routing:** Expo Router
* **Platforms:** iOS, Android, Web
* **State:** Zustand
* **Forms:** React Hook Form + Zod
* **UI:** NativeWind + shared design tokens
* **Media:** Expo camera / image picker / file APIs
* **Networking:** TanStack Query + typed API client

### Parent web app

* **Framework:** Next.js (App Router)
* **Styling:** Tailwind CSS
* **State/data:** TanStack Query
* **Charts:** Recharts

## 4.3 Backend

* **API service:** Next.js is not used as the backend runtime for MVP
* **Recommendation for MVP:** separate `apps/api` service using NestJS
* **Reason:** explicit backend boundaries, clearer ownership, easier async job and websocket support, less ambiguity for Codex

## 4.4 Data and storage

* **Primary database:** PostgreSQL
* **ORM:** Prisma
* **Cache/session/queue support:** Redis
* **Blob storage:** S3-compatible object storage
* **Search/log indexing:** defer until needed

## 4.5 Auth and identity

* **Authentication provider:** Clerk
* **Reason:** faster implementation of parent auth flows, session management, device handling, and re-auth patterns
* **Authorisation:** custom RBAC and household scoping in API service
* **Parent gate:** server-enforced re-auth or step-up challenge for sensitive actions

## 4.6 AI and content pipeline

### Local models

Use distinct local model classes by task:

* **Local text tutor model:** Gemma 4 E2B, quantized
* **Local multimodal model:** Gemma 3n E2B or E4B, feature-flagged for supported devices only

### Local runtimes

* **Mobile and supported web:** device-native or browser-compatible local runtime abstraction
* **Dev and desktop fallback:** llama.cpp-compatible runtime

Do not hard-bind the product to one local runtime implementation at the architecture level. Use an adapter boundary.

### Cloud orchestration

* backend service invokes stronger cloud models when routing rules require it
* no frontend app talks directly to cloud providers

### Moderation and policy checks

* dedicated safety service
* rule layer plus model-based classifiers
* structured outputs only

## 4.7 Observability

* **Logs:** structured JSON logs
* **Metrics/traces:** OpenTelemetry
* **Errors:** Sentry
* **Analytics:** PostHog

## 4.8 Testing

* **Unit:** Vitest
* **Component:** React Testing Library
* **E2E web:** Playwright
* **E2E learner app:** Maestro
* **Contract tests:** Vitest + Zod/OpenAPI validation

# 9. API Surface

Use **REST + OpenAPI**.

Reason:

* clearer contract generation
* cleaner separation between apps and API service
* less hidden coupling than tRPC
* better future interoperability

## 9.1 API namespaces

* `/auth/*`
* `/households/*`
* `/children/*`
* `/consent/*`
* `/learner-states/*`
* `/curriculum/*`
* `/sessions/*`
* `/stories/*`
* `/homework/*`
* `/safety/*`
* `/privacy/*`
* `/reports/*`

## 9.2 Example endpoints

### POST /households

Create household.

### POST /children

Create child profile.

### POST /consent/parental

Capture or update parent consent record.

### GET /children/:id/home

Return learner home payload:

* current streak
* recommended lesson
* pending reviews
* rewards snapshot

### POST /children/:id/baseline-assessment/start

Start baseline.

### POST /sessions

Start tutoring session.

Payload:

* childId
* mode
* subject
* optional curriculumNodeId

### POST /sessions/:id/turns

Submit child turn.

Payload:

* inputType: `text | image_reference | structured_response`
* content

### POST /sessions/:id/complete

Complete session and persist summary.

### POST /homework/parse

Upload or reference image and return extracted problem structure.

### GET /reports/children/:id/progress

Return parent dashboard progress data.

### GET /safety/events

List safety events by household.

### POST /safety/events/:id/review

Mark reviewed or record parent action.

### GET /privacy/children/:id/export

Request export package.

### POST /privacy/children/:id/delete

Request deletion workflow, subject to policy and legal constraints.

# 10. AI Orchestration Design

## 10.1 AI gateway rule

No frontend app talks directly to a cloud model provider.

Local inference is permitted through the learner app’s local runtime adapters, but authoritative policy checks, audit logging, and fallback decisions are enforced by feature policy.

## 10.2 AI routing modes

The AI router shall classify each request into one of four execution modes:

* `local_only`
* `local_preferred_cloud_fallback`
* `cloud_preferred_local_fallback`
* `cloud_required`

Routing decisions shall consider:

* device capability class
* runtime availability
* battery and thermal constraints when available
* task type
* required reasoning depth
* safety sensitivity
* network availability
* policy requirements
* feature flags

## 10.3 Task routing matrix

### Local-first tasks

* greeting and simple transitions
* lightweight explanations
* recap generation
* simple comprehension questions
* local draft story continuation
* first-pass intent classification
* first-pass confusion detection

### Local-preferred with cloud fallback

* ordinary guided tutoring turns
* baseline scoring
* routine homework help
* story generation under constraints
* learner-state summarisation

### Cloud-preferred

* parent-visible summaries
* progress narrative generation
* curriculum recommendation summaries
* reviewable explanations after repeated failures

### Cloud-required

* difficult multi-step reasoning
* difficult homework interpretation
* secondary safety review
* policy-sensitive child-facing responses after a safety trigger
* responses that failed local validation
* server-only workflows requiring stronger audit guarantees

## 10.4 AI subcomponents

### Tutor Orchestrator

Inputs:

* learner state
* curriculum target
* session mode
* recent transcript window
* safety context
* privacy and permission context
* device capability
* routing mode

Outputs:

* child-facing structured response
* state update proposals
* analytics events
* inference and fallback metadata

### Local Text Adapter

Responsibilities:

* load local text model
* expose typed inference calls
* report runtime health

### Local Multimodal Adapter

Responsibilities:

* handle image/text multimodal parsing where supported
* expose typed inference calls
* report capability limits

### Cloud Model Adapter

Responsibilities:

* call cloud providers
* enforce schemas and retries
* log model, version, and latency

### Content Generator

Produces:

* lesson variants
* story segments
* quizzes
* examples

Must emit schema-valid JSON only.

### Safety Classifier

Classifies:

* restricted topic risk
* emotional dependency language risk
* age-band mismatch
* unsafe instruction risk
* privacy-sensitive content risk

### Summariser

Produces:

* session summary
* parent-visible summary
* learner-state delta suggestion

## 10.5 Required structured outputs

Every AI response must conform to an explicit schema.

Example tutor response schema:

```ts
export const TutorResponseSchema = z.object({
  message: z.string(),
  messageStyle: z.enum(["coach", "story", "question", "feedback"]),
  suggestedUi: z.enum(["chat", "mcq", "free_text", "tap_choice", "story_panel"]),
  hints: z.array(z.string()).default([]),
  expectedResponseType: z.enum(["text", "choice", "number", "reading", "none"]),
  masterySignal: z.enum(["unknown", "improving", "mastered", "struggling"]).default("unknown"),
  shouldEscalateSafety: z.boolean().default(false),
  shouldRequireCloudReview: z.boolean().default(false),
  lessonStatePatch: z.record(z.any()).default({})
});
```

## 10.6 Session loop

1. Load child profile, permissions, learner state, and consent status.
2. Resolve lesson target.
3. Determine routing mode.
4. Build orchestration input object.
5. Generate candidate response locally if permitted.
6. Run validation and safety checks.
7. If invalid, low-confidence, or policy-sensitive, reroute to cloud or fallback response.
8. Persist turn metadata and inference metadata.
9. After child reply, update learner model.
10. Repeat until completion or interruption.

# 11. Safety Architecture

## 11.1 Safety layers

### Layer 1: Input classification

Classify incoming child text/image intent.

### Layer 2: Policy rules

Hard rules for forbidden content and routing.

### Layer 3: Model-based output review

Review candidate tutor response before delivery.

### Layer 4: Parent visibility and escalation

Create safety event and parent-facing review workflow when needed.

## 11.2 Safety event severities

* `info`
* `warning`
* `high`
* `critical`

## 11.3 Prohibited outputs

Do not allow output that:

* encourages secrecy from parents or trusted adults
* claims exclusive emotional bond
* gives self-harm instructions
* gives sexual content
* gives dangerous instructions
* gives diagnosis-like statements
* pressures the child into extended engagement
* manipulates reward or attachment behaviour

## 11.4 Safety policy versioning

Every safety decision must be traceable to:

* policy bundle version
* model deployment
* prompt template version
* inference run record

# 12. Privacy, Consent, and Retention

## 12.1 Required privacy controls

Primer must support:

* explicit parental consent records
* data minimisation by feature
* parent-visible data review
* export workflow
* deletion workflow
* retention scheduling
* child-safe defaults

## 12.2 Data minimisation rules

Do not collect or retain more than is required for:

* account operation
* learning progress
* safety review
* legal and abuse-prevention obligations

Avoid unnecessary behavioural telemetry in the learner app.

## 12.3 Parent controls

Parents must be able to:

* review child profiles
* review transcripts where policy allows
* review safety events
* change permissions
* request export
* request deletion

## 12.4 Retention policy

Retention must be policy-driven by data class, for example:

* session transcripts
* uploaded homework images
* safety events
* audit events
* learner progress data

Retention schedules must be stored and enforceable through jobs.

# 13. Realtime and Session Model

## 13.1 Session states

* `created`
* `active`
* `paused`
* `awaiting_input`
* `processing`
* `completed`
* `cancelled`
* `safety_paused`

## 13.2 Voice scope for MVP

Voice is **not a core MVP requirement**.

MVP interaction is:

* text-first
* image upload for homework help
* optional later milestone for voice

Do not architect around full duplex voice, interruption handling, or live spoken tutoring in MVP.

## 13.3 Realtime transport

For MVP:

* use HTTP for session steps
* use SSE only if needed for incremental updates

Do not introduce websocket complexity until there is a proven need.
